# PROBLEMA 1 — IDs DOS DRAGÕES FUNDIDOS ESTÃO INCONSISTENTES

Hoje o DragonTypeResolver gera IDs de fusão ordenando os elementos alfabeticamente.

Exemplo atual provável:

- fire + earth pode virar: fused_earth_fire
- water + earth pode virar: fused_earth_water
- fire + air pode virar: fused_air_fire
- water + air pode virar: fused_air_water
- water + metal pode virar: fused_metal_water

Mas a Taxonomia Dracônica em dragonTaxonomy.ts usa IDs diferentes, como:

- fused_fire_earth
- fused_water_earth
- fused_fire_air
- fused_water_air
- fused_water_metal

Isso faz o dragão nascer com um ID que a DragonScreen depois não reconhece corretamente via getDragonTypeById.

Critério:
Todo dragonType salvo em dragonData.dragonType deve existir em dragonTaxonomy.ts.

---

# CORREÇÃO OBRIGATÓRIA 1

Corrigir o DragonTypeResolver para usar um mapa explícito de combinações, em vez de gerar IDs só com sort alfabético.

Criar algo equivalente a:

const FUSED_TYPE_MAP: Record<string, string> = {
  'fire+water': 'fused_fire_water',
  'fire+earth': 'fused_fire_earth',
  'water+earth': 'fused_water_earth',
  'fire+air': 'fused_fire_air',
  'water+air': 'fused_water_air',
  'earth+air': 'fused_earth_air',
  'fire+metal': 'fused_fire_metal',
  'water+metal': 'fused_water_metal',
  'earth+metal': 'fused_earth_metal',
  'air+metal': 'fused_air_metal',
};

Para montar a chave, criar função segura que normalize a dupla sem depender do nome do ID:

function getPairKey(a, b) {
  const set = new Set([a, b]);
  if (set.has('fire') && set.has('water')) return 'fire+water';
  if (set.has('fire') && set.has('earth')) return 'fire+earth';
  if (set.has('water') && set.has('earth')) return 'water+earth';
  if (set.has('fire') && set.has('air')) return 'fire+air';
  if (set.has('water') && set.has('air')) return 'water+air';
  if (set.has('earth') && set.has('air')) return 'earth+air';
  if (set.has('fire') && set.has('metal')) return 'fire+metal';
  if (set.has('water') && set.has('metal')) return 'water+metal';
  if (set.has('earth') && set.has('metal')) return 'earth+metal';
  if (set.has('air') && set.has('metal')) return 'air+metal';
  return 'fire+water'; // fallback seguro, se necessário
}

Ou use uma implementação mais limpa, desde que o resultado final bata com os IDs reais de dragonTaxonomy.ts.

# PROBLEMA 2 — HATCHSCREEN PODE ENTRAR EM ERRO APÓS CONFIRMAR NOME

Na HatchScreen, após confirmar o nome, o código transforma o save:

hasEgg = false
eggData = null
hasDragon = true
dragonData = novoDragonData

Isso está correto.

Mas existe risco de a própria HatchScreen perceber eggData = null antes de navegar para DragonScreen e mostrar estado de erro por alguns instantes.

Isso acontece porque a tela tem uma verificação do tipo:

if (!egg && phase !== 'error') {
  setPhase('error');
}

Depois do nascimento completo, eggData realmente deve ser null.
Então a HatchScreen não pode tratar isso como erro durante a fase de finalização.

---

Ajustar a HatchScreen para não entrar em error depois que o jogador confirmou o nome e o nascimento está finalizando.

const isFinalizingRef = useRef(false);
